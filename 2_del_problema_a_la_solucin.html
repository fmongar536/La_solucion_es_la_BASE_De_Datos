2. Del problema a la solución

Antes de construir una base de datos, primero debemos entender qué problema queremos resolver. Esta fase es clave: si no analizamos bien los requisitos, es muy fácil que nuestra solución no funcione como debería. En este bloque serás como un analista de sistemas: observarás el problema, identificarás qué datos son necesarios, y empezarás a pensar cómo podrías organizarlos.

Pasarás por tres etapas: definir los requisitos, analizar la información relevante y hacer un primer esquema de cómo podría estructurarse tu base de datos. ¡Es el primer paso para construir una solución efectiva!

1. ¿Qué requisitos hemos de cumplir?

Antes de diseñar cualquier base de datos o aplicación, el primer paso es entender bien el problema que vamos a resolver. Para ello, debemos analizar con detalle qué necesita nuestro cliente, qué datos son importantes y qué funciones debe tener el sistema.

En esta fase de análisis de requisitos, el equipo de desarrollo trabaja junto al cliente para recopilar toda la información necesaria. El objetivo es redactar un documento claro y consensuado con los requisitos funcionales del sistema.

Vamos a practicar este proceso con un encargo simulado: una empresa necesita una aplicación para gestionar su agenda de clientes. Antes de la reunión con el cliente, vuestro equipo hace una lluvia de ideas con posibles funcionalidades. A continuación tienes una lista con esas propuestas. Lee con atención y selecciona únicamente aquellas que sean relevantes y coherentes con el encargo recibido. Desestima aquellas que no se ajusten al problema planteado.

Escenario del proyecto

Proyecto: Aplicación para gestionar una agenda de clientes de una empresa.

Funcionalidad principal: Almacenar datos de personas y consultarlos posteriormente.

Posibles requisitos del sistema

  • El programa será una agenda de clientes que permitirá guardar datos personales y consultarlos más adelante.
  • El programa pedirá un inicio de sesión.
  • Deberá mostrar, para cada persona, el nombre, los apellidos, el DNI, la fecha de nacimiento, el teléfono, el correo electrónico, la dirección postal y los servicios contratados.
  • Permitirá localizar a personas que cumplan años en el próximo mes.
  • Los datos obligatorios serán el DNI, el nombre, el primer apellido y el servicio contratado. El resto serán opcionales.
  • Permitirá grabar una cantidad elevada de datos.
  • Los datos se almacenarán en un fichero local para poder consultarlos en futuras sesiones.
  • Permitirá hacer búsquedas por cualquier dato personal: DNI, nombre o apellidos.
  • Permitirá buscar por tipo de producto para listar los clientes que lo han contratado.
  • Debe almacenar las últimas 10 búsquedas realizadas por el usuario.
  • El programa también gestionará los servicios ofrecidos por la empresa: se almacenará el código de la actividad y su descripción (ambos campos obligatorios).
  • Hará visibles los precios abonados por el cliente.

Tu tarea

Selecciona de la lista anterior los requisitos que consideras coherentes con el encargo. Justifica por qué los eliges y por qué descartas los demás.

Puedes presentar tu análisis como una tabla con dos columnas: “Requisito aceptado” / “Requisito descartado” y un breve motivo.

Este análisis será la base para diseñar tu base de datos más adelante, así que tómate tu tiempo para reflexionar con tu equipo.

¿Cuáles habéis seleccionado?

A tener en cuenta:

Nuestro programa estará instalado en la propia empresa donde solo hay un trabajador gestor de clientes, por lo que no necesita como requisito específico usar usuarios y contraseñas. Tampoco tiene la finalidad de bonificar a los clientes, por lo que no necesitamos saber su cumpleaños. La agenda de clientes no gestiona las facturas (podría plantearse como un módulo aparte en un futuro), así entonces no puede calcular importes abonados por los clientes (no es su principal objetivo). Finalmente, no necesitamos guardar las últimas búsquedas realizadas por el gestor de clientes, ya que no nos porporciona ningún beneficio adicional sobre la funcionalidad del software contratado por el cliente.
No obstante todas las funcionalidades especificadas, los detalles de los requisitos que puedan parecer no del todo concretos y las dudas se deben resolver en reuniones con el cliente. Esto permitirá afinar detalles y evitará malentendidos y posteriores exigencias de ampliación de funcionalidades, más allá de lo contratado.

2. Del qué al cómo: pensando como analistas

Ya conoces los requisitos generales del sistema. Sin embargo, a menudo una especificación poco precisa puede llevar a malentendidos técnicos o a implementar funciones que el cliente no necesita o no entiende de la misma forma.

En esta fase actuarás como un verdadero analista funcional: deberás realizar una lectura crítica y detallada de los requisitos definidos en la actividad anterior y detectar posibles ambigüedades que haya que aclarar antes de comenzar a diseñar o programar.

El objetivo es tomar decisiones claras sobre cómo debe comportarse el sistema en distintas situaciones. Esta fase puede marcar la diferencia entre una aplicación funcional y una confusa.

💡 ¿Sabías que...? El analista es la persona encargada de hablar con el cliente, observar cómo trabaja y formular las preguntas adecuadas para que el proceso de especificación de requisitos sea lo más correcto y detallado posible.

Actividad: Resolviendo ambigüedades

Responde con precisión a las siguientes preguntas. Algunas pueden tener múltiples opciones válidas, pero deberás justificar tu elección. Recuerda: tus respuestas influirán directamente en el diseño de la base de datos y el comportamiento del programa.

  1. ¿Sólo se podrán visualizar datos realizando una búsqueda por datos personales?
    ¿O también por otros criterios?
  2. ¿Qué datos se mostrarán de cada cliente cuando se realiza una búsqueda? ¿Todos los campos? ¿Solo algunos?
  3. ¿Las búsquedas serán exactas o se permitirá buscar por fragmentos? (Ej. buscar "Gonz" para encontrar "González")
  4. ¿Los datos se guardarán automáticamente o tras pulsar una opción específica?
  5. ¿Se permitirá modificar o eliminar datos ya guardados?
  6. ¿Cómo se dará de alta un nuevo servicio de la empresa? ¿Dónde se introducen esos datos? ¿Qué campos son obligatorios?
  7. ¿Se podrán eliminar servicios? ¿Cómo?
  8. ¿El código del servicio se generará automáticamente o lo decidirá el usuario? ¿Qué formato tendrá?
  9. ¿Cómo se mostrará la información por pantalla? ¿Una sola línea por cliente, varias líneas, por campos separados…?
  10. ¿La búsqueda por tipo de producto será por código, por nombre o por ambos?

Aplicación a la realidad

Es importante pensar en cómo podremos aplicar todo lo que aprendemos a nuestra vida real, a nuestro día a día. Así, trasladando el proceso que acabamos de presentar, en la realización de un proyecto real es habitual volver varias veces a la fase de análisis, por eso es deseable realizar varias reuniones con el cliente entre fase y fase para que las necesidades y carencias puedan detectarse rápidamente. Por ejemplo: Si el proyecto durase varios meses, son deseables reuniones cada dos semanas ya que esto evitará costes adicionales de tener que volver atrás por malentendidos.

3. Esquema visual del sistema: primer diseño del programa

A veces, los documentos técnicos con descripciones extensas no son fáciles de entender para todos los perfiles, especialmente si no tienen formación en informática. Por eso, una excelente forma de comunicar qué hace un sistema es mediante un diagrama de casos de uso.

Este tipo de diagrama nos permite representar de forma visual, sencilla y estructurada qué funciones ofrece un sistema y quién puede realizarlas. Es una herramienta muy usada en la fase de análisis, tanto en entornos profesionales como educativos.

¿Qué es un diagrama de casos de uso?

Un diagrama de casos de uso es un esquema que representa:

  • Los actores: personas o sistemas que interactúan con la aplicación.
  • Las acciones (casos de uso): las funciones que pueden realizar en el sistema.
  • El sistema: representado como un rectángulo que contiene todas las acciones disponibles.

Se utiliza habitualmente para mostrar de forma simple cómo debe funcionar un programa desde el punto de vista del usuario, especialmente en las primeras fases del proyecto.

Tu tarea

Crea un diagrama de casos de uso que represente el sistema que estás diseñando para gestionar la agenda de clientes. Sigue estos pasos:

  1. Dibuja un rectángulo que represente el sistema completo (la aplicación).
  2. Dentro, coloca elipses con las acciones principales (buscar cliente, añadir datos, modificar, eliminar, etc.).
  3. Fuera del rectángulo, dibuja los actores (por ejemplo, administrador, usuario...).
  4. Conecta cada actor con las acciones que puede realizar mediante líneas.

Para hacer el diagrama debes usar una herramienta digital como draw.io (diagrams.net).

Debes entregar:

  • ✅ Una imagen del diagrama.
  • ✅ Una explicación breve (máx. 150 palabras) del funcionamiento general del sistema y de los actores que has incluido.

¿Quieres ver un ejemplo resuelto?

Ejemplo resuelto: Diagrama de casos de uso de un restaurante

Para ayudarte a entender mejor qué se espera en esta actividad, aquí tienes un ejemplo resuelto de un sistema simplificado sobre un restaurante.

Este diagrama muestra los actores principales (el chef y el crítico de comidas) y las acciones que pueden realizar dentro del sistema.

Observa cómo:

  • El sistema está representado como un rectángulo que contiene las acciones.
  • Las acciones (casos de uso) están escritas en elipses, como "Probar la comida" o "Pagar la comida".
  • Los actores están fuera del sistema (chef y crítico) y conectados a las acciones que pueden realizar.

Este tipo de esquema es muy útil para visualizar el comportamiento de un sistema sin necesidad de escribir código. También permite que personas no técnicas (clientes, responsables de proyecto…) entiendan qué hace exactamente una aplicación antes de desarrollarla.

Tú deberás hacer algo similar con tu propio proyecto. Este esquema te servirá como base para diseñar más adelante el modelo de datos con entidades y relaciones.

Es un diagrama que describe el comportamiento que tiene el software en respuesta a eventos realizados por actores externos.

Algunas pistas

Ten en cuenta a la hora de hacer el diagrama para nuestro cliente:

  • Necesitaremos representar al empleado que se encargará de los clientes y al administrador del sistema.
  • El administrador podrá realizar las mismas operaciones que el cliente pero además deberá poder borrar clientes y servicios de la empresa. ¿Debería poder hacerlo también el empleado?

¿Y ahora qué?

Después de analizar el problema, definir los requisitos y representar las funcionalidades del sistema, ya estás en condiciones de identificar la herramienta clave que hará posible todo esto: las bases de datos.

Será el motor que permitirá almacenar, organizar y consultar la información que gestionará tu aplicación. A partir del siguiente bloque vas a aprender a crear bases de datos relacionales, diseñar sus estructuras y manipular la información con SQL.

Prepárate, porque el siguiente paso te permitirá construir una solución sólida, eficaz y... 100% funcional. ¡Vamos a por ello!
