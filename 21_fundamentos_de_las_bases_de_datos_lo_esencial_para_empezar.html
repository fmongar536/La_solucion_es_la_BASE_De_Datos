2.1 Fundamentos de las bases de datos: lo esencial para empezar

1. ¿Por qué no basta con archivos?

Durante años, muchas aplicaciones han almacenado datos usando archivos de texto o ficheros simples (como .txt, .csv, etc.). Aunque en algunos casos pueden funcionar, cuando el volumen de datos crece o hay varios usuarios trabajando al mismo tiempo, estos métodos empiezan a fallar.

Aquí tienes algunos problemas típicos que aparecen al trabajar solo con archivos:

  • ❌ Duplicidad de datos: se repite la misma información en varios archivos.
  • ❌ Inconsistencias: los datos no coinciden entre archivos porque alguien los ha modificado solo en uno.
  • ❌ Dificultad para acceder a los datos: no hay una forma rápida de consultar o filtrar información concreta.
  • ❌ Falta de control de acceso: cualquier persona puede modificar los archivos, incluso por error.
  • ❌ Escasa automatización: no se pueden hacer consultas complejas ni relaciones entre datos fácilmente.

Por eso, cuando queremos diseñar un sistema sólido, compartido y escalable, la solución ideal es usar un Sistema de Gestión de Bases de Datos (SGBD).

2. Todo bajo control: cómo funciona una base de datos

Como has visto en el apartado anterior, los sistemas basados en archivos presentan muchos problemas cuando gestionamos grandes cantidades de información o cuando varios usuarios acceden a los datos a la vez.

Por eso surgieron las bases de datos: estructuras organizadas donde se pueden almacenar, modificar y consultar datos de forma eficiente, minimizando la redundancia y los errores.

Una base de datos es un conjunto de datos organizados que pueden ser compartidos por múltiples programas y usuarios, según los permisos de acceso establecidos. De ahí el término "base": porque actúa como un punto común para la información.

¿Y quién gestiona todo eso?

Las bases de datos funcionan gracias a un software especializado: el Sistema Gestor de Bases de Datos (SGBD).

Un SGBD es el conjunto de programas que permite:

  • 📂 Crear, modificar y borrar bases de datos.
  • 🔐 Controlar quién accede a los datos y qué puede hacer con ellos.
  • 🔄 Evitar errores, duplicidades y accesos simultáneos conflictivos.
  • 📊 Consultar datos mediante lenguajes de consulta como SQL.

Algunos ejemplos de SGBD muy conocidos son: MySQL, PostgreSQL, SQLite, Oracle o Microsoft SQL Server.

Actividad: ¿Qué sistema es mejor?

Lee cada afirmación y, antes de desplegar la respuesta, piensa si se adapta mejor a un sistema basado en archivos o a uno basado en bases de datos.

1. Permite que varios usuarios accedan a los datos a la vez sin errores ✅ Base de datos. Los SGBD gestionan el acceso concurrente de forma controlada.
2. No tiene control de seguridad ni permisos de usuario ❌ Archivos. Cualquiera puede abrir y modificar el contenido.
3. Permite hacer búsquedas por diferentes criterios fácilmente ✅ Base de datos. Gracias al lenguaje SQL puedes consultar cualquier dato rápidamente.
4. Tiene riesgo de duplicar datos en distintos archivos ❌ Archivos. No hay control automático de integridad.
5. Relaciona información entre distintas tablas fácilmente ✅ Base de datos. Es una de las grandes ventajas del modelo relacional.

Esquema de una Base de Datos (BD)

¿Qué se almacena?

Hay que mirar desde dos puntos de vista:
Desde un punto de vista lógico
(el de los usuarios):
  • Se almacenan todos los datos en un solo lugar.
  • Se almacenan las definiciones de las relaciones entre los datos.

Desde un punto de vista físico:
  • Los datos se almacenan en uno o varios discos y en uno o varios archivos.
Así, la parte lógica y la parte física son independientes.

¿Cómo se accede?

A los datos se accede mediante el sistema de gestión de la base de datos (SGBD) que permite 3 formas de ver los datos:

  • Esquema de la base de datos: Visión lógica global para el administrador y el diseñador.
  • Subesquema o Vista: Visión parcial de los datos relacionados con una aplicación o con la función que tenga asignada el usuario.
  • Visión física: El modo físico en que están almacenados los datos para los programadores creadores de la BD.



Sistema Gestor de Bases de Datos (SGBD)

También conocido por sus siglas en inglés DBMS (DataBase Management System).

¿Para qué sirve un DBMS?

Es una aplicación que:

  • Permite definir y crear la base de datos gracias a un lenguaje de definición de datos (DDL).
  • Permite trabajar con los datos mediante un lenguaje de manejo de datos (DML).
  • Ofrece un acceso controlado a los datos gracias a que tiene:
    • Un sistema de seguridad.
    • Un sistema de integridad.
    • Un sistema control de concurrencia.
    • Un sistema de control de recuperación.
    • Un diccionario de datos, donde almacena la descripción de los datos y gracias a esto es independiente del almacenamiento físico de los mismos.

3. Cada dato en su sitio... y todos conectados

Ahora que ya sabes qué es una base de datos y qué papel juega el SGBD, ha llegado el momento de conocer cómo se organizan internamente los datos.

En esta unidad vamos a trabajar con bases de datos relacionales, que son las más utilizadas en el mundo real. Este modelo organiza los datos en forma de tablas, que se relacionan entre sí mediante claves.

Cada tabla representa un conjunto de elementos del mundo real (por ejemplo, clientes, productos o reservas), y cada fila representa una tupla o registro. Las relaciones entre esas tablas se establecen gracias a claves primarias y claves ajenas.

Elementos clave de una base de datos relacional

  • 📄 Tabla: conjunto de datos relacionados. Cada tabla tiene un nombre único.
  • 📌 Atributos: columnas que definen las características de los datos (nombre, fecha, email, etc.).
  • 📎 Tupla: cada fila representa un registro único (una instancia real del dato).
  • 🔑 Clave primaria (PK): atributo(s) que identifica de forma única a cada tupla. No puede repetirse.
  • 🔗 Clave ajena (FK): atributo que enlaza una tabla con otra. Debe coincidir con la PK de la tabla referenciada.

Comprensión del modelo relacional

1. ¿Qué columna identifica de forma única cada oficina en esta base de datos? ✅ La combinación de ENTIDAD + CODIGO_OFICINA. Es una clave compuesta que garantiza unicidad incluso cuando diferentes entidades usan el mismo código de oficina.
2. ¿Qué papel cumple la columna ENTIDAD en la tabla OFICINA? ✅ Es una clave ajena (FK). Sirve para relacionar cada oficina con su banco correspondiente a través de la clave primaria ENTIDAD de la otra tabla.
3. ¿Qué problema podría surgir si se usara solo CODIGO_OFICINA como clave primaria? ❌ Podría haber conflictos si varias entidades usan el mismo código. ✅ La clave dejaría de ser única, lo que rompería la integridad del modelo.
4. ¿Qué ventaja tendría crear un campo ID_OFICINA como clave primaria? ✅ Facilita relaciones, mejora el rendimiento y permite cambios en los códigos sin afectar otras tablas. Se convierte en una clave artificial (sustitutiva) más eficiente y mantenible.
5. ¿Qué tipo de relación hay entre ENTIDAD y OFICINA? ✅ Relación 1:N (uno a muchos). Un banco puede tener varias oficinas, pero cada oficina pertenece a una sola entidad.
