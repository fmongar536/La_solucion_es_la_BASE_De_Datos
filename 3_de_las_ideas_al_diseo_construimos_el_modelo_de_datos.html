3. De las ideas al diseño: construimos el modelo de datos

¡Buen trabajo hasta aquí! Has completado con éxito la fase de planificación y análisis del proyecto. Ahora comienza una nueva etapa del ciclo de vida del software: el diseño.

En este punto vamos a preparar cómo se organizarán los datos y cómo se estructurará internamente el sistema. Para ello, usaremos herramientas llamadas modelos de datos, que nos ayudarán a representar visualmente tanto la información como las relaciones entre los elementos.

Actividad: ¿Qué modelo es cuál?

Existen varios niveles de modelado de datos: conceptual, lógico y físico. Cada uno cumple una función específica dentro del diseño de una base de datos.

Antes de entrar en detalle, pon a prueba tu capacidad de deducción: observa las siguientes imágenes y trata de relacionar cada una con el tipo de diseño que representa.

  • Modelo conceptual: Es la representación más abstracta y general. Describe la información del sistema usando entidades (como Cliente o Producto), atributos (como nombre o precio) y relaciones entre ellas. No se usa lenguaje técnico ni se mencionan tipos de datos. Se representa mediante diagramas entidad-relación (E/R).
  • Modelo lógico: Traduce el modelo conceptual a una estructura lógica que pueda implementarse en un SGBD. Aquí ya aparecen las tablas, claves primarias, claves ajenas, tipos de datos, etc., pero sin entrar aún en cómo se almacenan físicamente. Este modelo depende del tipo de base de datos que se vaya a usar (relacional, NoSQL, etc.).
  • Modelo físico: Representa cómo se almacenarán realmente los datos en disco. Se definen los nombres reales de tablas, los índices, formatos de almacenamiento, y se optimiza para mejorar el rendimiento del sistema. Este modelo ya está 100% ligado al software y al hardware.

¿Serás capaz de identificar correctamente el modelo conceptual, el lógico y el físico?
